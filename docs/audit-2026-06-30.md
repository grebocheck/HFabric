# HFabric — Code-Quality & Stability Audit (2026-06-30)

Scope: a follow-up to the [2026-06-14 release-readiness audit](audit-2026-06-14.md),
re-run after **P25** (unified Model Manager + CivitAI), **P26** (Edit workspace), and
**P27** (video workspace) landed and the project tagged through **v0.3.0**. This pass
asks a different question than the last one: not *"can it be handed to testers?"* (it
has been — three releases shipped) but **"is the codebase healthy and stable enough to
keep changing safely?"**

Method: ran every CI gate locally, read the load-bearing concurrency/memory code
(`arbiter.py`, `scheduler.py`), measured per-module coverage, sized every source file,
and swept exception-handling and type-safety patterns. Every finding below cites a
file and a number so it is checkable.

## Verdict

**Overall: B+ / "healthy core, velocity debt at the edges".** The architecture that
earned an A− last time is intact and, if anything, better factored (the image backend
grew a `image_diffusers_parts/` package; `types.ts` now derives from OpenAPI-generated
types instead of mirroring by hand). All gates are green. Nothing here is a
user-visible bug.

What changed is **debt accrued by shipping five workstreams back-to-back**: the
coverage floor had become a tripwire rather than a safety net, several services lacked
focused tests, a few files had grown past safe-review size, and an in-flight video
change sat uncommitted and under-tested. None of it was on fire; all of it made the
*next* change riskier. Hence the roadmap now leads with a quality track (Q1–Q7).
Follow-up work on the same pass has already landed Q1, Q2 backend splits, Q3, Q5, Q6,
Q7, CLI real-GPU validation for LTX/Wan video, the live app-path video smoke
(HTTP range replay, cancel, Video -> LLM -> Video swap), P27.6 UI polish
(richer presets + History filters), P24.6 invite-readiness (README first impression
+ GitHub metadata), and Q4 landing. The remaining risk is concentrated in
hardware-dependent release breadth.

## Findings (ranked by risk-reduction value) — maps to roadmap Q1–Q7

### Q1 — The coverage floor was a tripwire, not a net (resolved)
Initial finding: `--cov-fail-under=63` vs an actual **63.44%** left a **0.44-point**
margin. Follow-up on 2026-06-30 added focused tests and raised CI to
`--cov-fail-under=68`; local verification is **426 passed / 69.08%**.

The original lowest-service gaps were:

| Module | Coverage | Statements missed |
|--------|---------:|------------------:|
| `services/accelerator_runtime.py` | 28.6% | hardware probe (harder to test) |
| `services/chat_attachments.py` | 31.8% | 152 / 238 — **decode/validation is security-adjacent** |
| `services/embedding_service.py` | 37.2% | 51 / 89 |
| `services/video_service.py` | 40.6% | 13 / 26 |
| `services/rag_service.py` | 55.8% | 15 / 38 |
| `services/chat_service.py` | 57.9% | 18 / 60 |

**Done:** added service coverage for `chat_attachments`, `chat_service`,
`rag_service`, `video_service`, `embedding_service`, plus adjacent
`accelerator_runtime`, `events`, `pidfiles`, `queue_service`, `videos` API, and
scheduler recovery coverage.

### Q2 — Backend monoliths were too large to review safely (resolved)
Follow-up split the high-churn mode dispatch into
`backends/image_diffusers_parts/generation.py`; `_generate_real` now delegates to the
parts package and keeps metadata/persistence locally. New dispatcher tests cover
txt2img, img2img, inpaint, and ControlNet branches across the relevant families. The
next backend tier was split too: `settings_overrides.py` is now a 159-line runtime
load/save/sanitize module backed by `settings_specs.py`, and `voice_engine/realtime.py`
is now a 561-line session/stream module backed by `realtime_processor.py`.

### Q3 — Frontend monoliths were too large to review safely (resolved)
Follow-up moved `ChatPanel` send/edit/attachment/manual-image workflow actions into
`ChatPanelHooks.ts` and moved the `VoicePanel` Live Console / Tuning / Routing /
Presets / Offline Convert / Diagnostics sections into `VoicePanelSections.tsx`.
Current sizes: `VoicePanel.tsx` **985**, `ChatPanel.tsx` **854**. `types.ts` (946) is
**no longer** a concern — it derives from `types.generated.ts`.

### Q4 — In-flight P27 video work is now tested and landed (resolved)
The working tree carries new `_FAMILY_GEN_DEFAULTS`, bnb `model`/`sequential`
offload-branch selection, `_bounded_float`, and `_DENOISE_END`/`_ENCODE_START` progress
phases in `video_diffusers.py` plus `VideoComposer.tsx` changes. Follow-up added stub
coverage for family defaults, bnb offload branch selection, guidance clamping, and
decode/encode progress headroom. CLI real-GPU smoke passed on 2026-06-30 for LTX T2V,
LTX I2V, and Wan T2V at 832x480 / 49 frames / 8 steps; the LTX I2V pass caught and
fixed a VAE dtype mismatch. The live app-path smoke also passed via
`scripts/video_app_smoke.py`, covering websocket events, HTTP range replay, cancel
during denoise, and Video -> LLM -> Video resident swap. The WIP was landed as part
of this quality pass. A follow-up in the same pass added the FramePack Hunyuan
composite code path (`framepack-hunyuan-i2v/{base,transformer,redux}`), I2V-only
API/UI guards, bnb 4-bit loader coverage, and download catalog entries. The local
repos were fetched and a real-GPU FramePack I2V smoke passed at 480x832 / 91
requested frames / 8 steps, producing 109 output frames at a 9.67 GB VRAM peak.

### Q5 — A handful of silent exception swallows (resolved)
278 `except` blocks across 52 files; the overwhelming majority are deliberate
best-effort, annotated `# noqa: BLE001 - <reason>`. The exceptions that swallow with a
bare `pass` and **no log** are in load-bearing spots: `core/events.py` L44/L61 (the
event bus) and `voice_engine/realtime.py` L597/L608 (stream teardown). For the
"debuggable after a crash" objective these should `logger.debug(..., exc_info=True)`.
Follow-up added debug logging with reason comments and event-bus tests.

### Q6 — Load-bearing safety paths need named tests (resolved)
The arbiter/scheduler recovery branches *are* the product promise. Confirm and fill
named STUB tests for: RAM-budget refusal + warm-evict ladder (`arbiter._guard_budget`),
resident-pin park/resume (`scheduler._pick_next`), orphan requeue on restart
(`scheduler._requeue_orphans`). Follow-up confirmed the RAM guard tests and added
resident-pin plus orphan-requeue tests.

### Q7 — Roadmap/audit truth drift (low, but corrosive)
The roadmap's "Next up #1" still read *"push the first `v0.1.0` tag = the launch"* while
`v0.1.0`/`v0.2.0`/`v0.3.0` were all tagged and shipped, and the prior audit predated
P25/26/27. **Fixed in this pass** (roadmap rewritten, this doc added). Adopt a standing
rule: refresh the audit snapshot and prune the roadmap on every minor release.

## What is genuinely healthy (don't disturb)

- **Arbiter** (`core/arbiter.py`): the one-resident invariant, swap/keep-warm/pin/lane
  logic, and the pre-load RAM budget guard are clear, well-commented, and correct.
- **Scheduler** (`core/scheduler.py`): phase-batching shared between the live worker
  and the plan preview (so predicted swaps can't drift), structured `logger.exception`
  on job failure, orphan requeue on restart, and a `finally` that always records the
  memory profile and runs cleanup.
- **Frontend type safety**: `types.ts → components["schemas"]` over generated types;
  6 casts and 0 stray `console.*` across non-test source.
- **Data layer**: Alembic migrations real and current; `session_scope` commits/rolls
  back correctly.
