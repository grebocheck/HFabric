# HFabric — Release-Readiness Audit (2026-06-14)

Scope: a follow-up to the [2026-06-11 audit](audit-2026-06.md), focused on one
question: **what stops this from being handed to other people to test?** It
re-reads the same surface (backend, frontend, scripts, CI, docs, repo hygiene)
after the P14–P20 phases landed, and grades the project against *external-tester*
readiness rather than internal correctness.

Both suites still run on every push (stub mode, no GPU): backend pytest with a
coverage floor, frontend `tsc -b` + vitest + `npm run build`, plus ruff and
eslint lint gates.

## Verdict

**Overall: B+ (8.0/10), but "internal-use solid, external-test rough".** The
2026-06 audit's four critical/high risks (no auth, hand-rolled migrations, no
backup/logs, dev-only run story) are **resolved**. The app now has bearer-token
auth, Alembic migrations, rotating logs, a backup script, a production serving
mode, a capability-profile system that gates features per GPU, and a managed
llama.cpp runtime. That is real, shipped progress.

What remains is almost entirely **release engineering and breadth of
validation**, not core correctness:

| Area | Then (06-11) | Now (06-14) | Note |
|------|------|------|------|
| Architecture | A− | A− | Arbiter/sysmon invariants unchanged and still excellent |
| Memory discipline | A | A | Core promise; learned profiles + guards intact |
| Security | D | B+ | Token auth, loopback-gated desktop actions, sane defaults |
| Data layer | C+ | A− | Alembic, backups, rotating logs all shipped |
| Distribution | C | B− | Prod serving + one-command launcher shipped; **no versioned/packaged release yet** |
| Tests & CI | B | B+ | Coverage floor + lint + build in CI; routers now covered; GPU paths still untestable in CI |
| Code health | B− | B− | **VoicePanel grew to 1651 lines**; the P17 splits are still all open |
| Cross-platform | (n/a) | C | NVIDIA/Windows validated; **ROCm and MPS implemented but never run on real hardware** |
| Docs | A− | B | Excellent but drifting: stale test counts, Windows-centric residue, a 678-line README that mixes user + dev reference |

## What got fixed since the last audit (keep it)

- **W1 (no auth) → fixed.** `HFAB_API_TOKEN` bearer middleware, WebSocket
  `?token=` gate, desktop-reaching endpoints (`reveal_image`) hard-gated to
  loopback regardless of token, and a default bind of `127.0.0.1`. The threat
  model is documented. For a local single-user app this is now appropriate.
- **W3 (migrations) → fixed.** Alembic with a baseline revision; startup runs
  `upgrade head`; legacy raw-SQL rows covered by tests.
- **W7 (no logs) → fixed.** Rotating `data/logs/hfabric.log`; job lifecycle,
  arbiter notes, and unhandled exceptions are persisted for post-mortems.
- **Backup story** → `scripts/backup.py` with the SQLite backup API + retention.
- **W8 (dev-only) → fixed.** `HFAB_SERVE_FRONTEND=true` serves the built SPA from
  one FastAPI port; `--prod` launcher path builds, waits on health, opens browser.
- **Universal install (P20)** → hardware probe, profile resolver, capability
  profile, per-model compatibility gating, Setup Doctor, managed llama.cpp
  runtime. This is the headline feature delivered since the last audit.

## Release-readiness gaps (ranked by risk to an external tester)

### R1 — The multi-platform story is validated on exactly one machine (high)
The whole P20 effort makes the app *claim* NVIDIA CUDA, AMD ROCm (Linux), and
Apple MPS support. Only **NVIDIA / RTX 5070 Ti / Windows 11** has ever been run
end-to-end. `docs/gpu-smoke.md` carries explicit `TODO` rows for ROCm and MPS.
The resolver logic is unit-tested with fake probes, but no one has confirmed a
real ROCm or Apple Silicon box loads a model and generates an image. For
external testers, a non-NVIDIA user is running an unproven path. **Mitigation:**
either recruit ROCm/MPS testers explicitly and label those profiles "experimental
— help us validate", or scope the first external release to NVIDIA only and say so.

### R2 — Voice live session (resolved 2026-06-14)
Originally flagged because the realtime engine had only *synthetic* benchmarks.
**The user has since validated it with a real microphone:** live conversion +
monitor output work well, and a queued image/LLM job parks during a session and
resumes after. P6R.4 is closed. The remaining caveat is code-health, not
correctness: voice is still the largest body of code (`voice_engine/` package +
the 1651-line panel — see R3). Recording exact round-trip latency numbers in
`gpu-smoke.md` stays an optional follow-up.

### R3 — `VoicePanel.tsx` is now the worst monolith (1651 lines) (medium)
The 06-11 audit flagged it at 749 lines; the P6R rework **doubled it**. The other
flagged monoliths are unchanged: `image_diffusers.py` (1461),
`ChatPanel.tsx` (1070), `ImageComposer.tsx` (732), `Gallery.tsx` (632),
`types.ts` (825, hand-maintained). None of the P17 split tasks have started.
This is feature-velocity debt, not a user-facing bug, but every voice/image
change now lands in a file too large to review safely.

### R4 — Documentation has drifted out of sync with the code (medium)
Trust-eroding for a new reader:
- README and ROADMAP say **"143 tests green"**; the suite is now ~248 backend
  test functions + ~66 frontend cases. The 06-11 audit says "backend 107".
- ROADMAP marks **P16.1 (coverage in CI)** and **P16.3 (frontend lint in CI)** as
  `[~] in progress`, but CI already runs `--cov-fail-under=63` and `npm run lint`
  with committed eslint/prettier configs. They are done.
- README's clone step hardcodes `cd d:\VSCode\ImageFabric` — the author's own
  path, not a generic instruction.

### R5 — Build artifact still tracked in git (low, but it's noise on every PR)
`frontend/tsconfig.tsbuildinfo` is committed and shows up modified in `git status`
on every build (P17.6, still open). It churns every diff and is exactly the kind
of noise a first-time contributor trips on.

### R6 — Onboarding a model is still CLI-only (medium for non-technical testers)
A fresh user with no models sees "No image models discovered". The Setup Doctor
shows *what would* be downloaded, but the actual download is `huggingface-cli` /
`python scripts/fetch_models.py` from a terminal. The in-app **model download
manager (P18.4)** is the missing piece that turns "clone + read README + run CLI"
into "click Recommended → download". For non-developer testers this is the
biggest friction wall.

### R7 — No release engineering yet (medium)
There is no version number anywhere (no `__version__`, no git tag, no
`package.json` version of record), no `CHANGELOG.md`, no `CONTRIBUTING.md`, no
issue templates, and no packaged artifact — testing means "clone the repo". For
an external test round you want: a version stamp surfaced in `/api/health` and the
UI, a changelog so testers know what changed between drops, and a one-page "how to
report a bug" so feedback is actionable.

### R8 — Windows-centric residue in user-facing surfaces (low-medium)
The docs and a few panels still assume Windows: README references
`bin/llama/llama-tts.exe` / `llama-mtmd-cli.exe` by their `.exe` names, and the
hardcoded dev path in R4. The managed llama runtime handles cross-platform binary
names internally, but the docs read as Windows-only, which undercuts the
multi-platform message. **Mitigation:** describe binaries by role, not filename;
keep platform-specific commands clearly labeled.

### R9 — Backend↔frontend type drift remains hand-managed (low)
`types.ts` (now 825 lines, up from 515) still mirrors backend schemas by hand
(P17.4 OpenAPI codegen not done). Not urgent, but every new endpoint is a chance
to ship a silent shape mismatch that only surfaces at runtime in a tester's
browser.

## Recommended sequencing toward a first external test build

1. **Truth-in-docs pass** (R4, R5, R8) — cheap, immediate. Fix counts, stale
   markers, the hardcoded path, the tracked build artifact. *(This audit ships
   with the README/ROADMAP rewrite that does most of it.)*
2. **Decide the platform scope of v0.1** (R1) — recommend NVIDIA-only "supported",
   ROCm/MPS "experimental, help wanted". Put it in the README support matrix.
3. ~~**Finish or fence the risky features** (R2)~~ — **done**: P6R.4 closed, voice
   validated with a real mic on 2026-06-14.
4. **Release engineering** (R7) — version stamp + `CHANGELOG.md` +
   `CONTRIBUTING.md` + a "report a bug" path. Tracked as **P21** in the ROADMAP.
5. **Onboarding** (R6) — the P18.4 model download manager is the single highest-
   leverage UX item for non-developer testers.
6. **Then** resume code-health splits (R3, R9 / P17) as feature work demands.

The full remediation plan with checkboxes lives in [ROADMAP.md](../ROADMAP.md)
(release-readiness phase **P21**, plus the still-open P17/P18.4/P19 items).
